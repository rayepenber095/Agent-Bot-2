PART 0 — WHO THIS IS FOR AND WHAT YOU ARE BUILDING
═══════════════════════════════════════════════════════════════
Purpose
You are building VulnLab Pro — a self-hosted, intentionally vulnerable
web application designed for cybersecurity students, ethical hackers, CTF
players, bug bounty hunters, and penetration testers to practice real-world
attack and defense techniques in a completely safe, local environment.
This application is inspired by:

Slack → channels, real-time messaging, file sharing
Amazon → product listings, cart, checkout, orders
Social platforms → profiles, posts, comments, follows
Online banking → wallet, transfers, transaction history

Combined, these modules create a realistic target environment that mirrors
the complexity of real enterprise applications.
Who will use this?

Penetration testers learning to find OWASP Top 10 vulnerabilities
Developers learning secure coding by comparing vulnerable vs. secure code
Students taking OSCP, CEH, eJPT, or similar certification courses
CTF players wanting a local realistic practice target
Trainers running cybersecurity workshops

What makes this different from DVWA?
Unlike DVWA (which is simplistic and outdated), VulnLab Pro:

Has a realistic multi-module application (not just single form demos)
Uses a modern stack (GraphQL, React, MySQL, JWT, Socket.io)
Has a one-click toggle that switches the ENTIRE app's security posture
Includes production-quality logging so learners can see what defenders see
Has 10 default users with realistic data to practice IDOR, enumeration, etc.
Includes a banking system to practice financial logic flaws
Documents every vulnerability with exploitation steps

Use mysql2 with RAW SQL string concatenation in vulnerable mode (e.g.,
"SELECT * FROM users WHERE username='" + username + "'") so that SQL
injection is REAL and exploitable. In hardened mode, switch to parameterized
prepared statements using ? placeholders. This is the core of the A03
injection demonstration and must be genuinely exploitable.

1.3 Why Raw SQL Instead of an ORM?
Using Prisma or Sequelize would PREVENT genuine SQL injection because
these tools automatically parameterize queries. VulnLab Pro MUST use raw
mysql2 queries so that learners can run actual ' OR '1'='1 payloads,
use sqlmap, and see real UNION-based extraction. The educational value
is lost if queries are auto-sanitized. In hardened mode, the same endpoints
use ? placeholders which is the actual fix.

sql-- ─────────────────────────────────────────────────────────────
-- DATABASE: vulnlab

CREATE DATABASE IF NOT EXISTS vulnlab CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE vulnlab;

-- ─────────────────────────────────────────────────────────────
-- TABLE: users
-- Purpose: All registered users including admins
-- Vuln notes:
--   password_plain: always stores plaintext (for A02 crypto failure demo)
--   secret_note: contains fake sensitive data (for IDOR demo A01)
--   ssn: fake social security number (for IDOR demo)
-- ─────────────────────────────────────────────────────────────
CREATE TABLE IF NOT EXISTS users (
  id                INT AUTO_INCREMENT PRIMARY KEY,
  username          VARCHAR(50) UNIQUE NOT NULL,
  email             VARCHAR(100) UNIQUE NOT NULL,
  password          VARCHAR(255) NOT NULL,      -- bcrypt in hardened, plaintext in vulnerable
  password_plain    VARCHAR(100) NOT NULL,       -- ALWAYS plaintext for A02 demo
  role              ENUM('user','moderator','seller','admin','sudo') DEFAULT 'user',
  full_name         VARCHAR(150),
  avatar_url        VARCHAR(500) DEFAULT '/uploads/avatars/default.png',
  bio               TEXT,
  phone             VARCHAR(20),
  address           TEXT,
  ssn               VARCHAR(15),                 -- fake SSN e.g. "123-45-6789" for IDOR
  dob               DATE,
  secret_note       TEXT,                         -- fake API keys, secrets for IDOR
  balance           DECIMAL(15,2) DEFAULT 1000.00,
  is_active         BOOLEAN DEFAULT TRUE,
  is_locked         BOOLEAN DEFAULT FALSE,        -- lockout after failed logins (hardened mode)
  failed_logins     INT DEFAULT 0,
  last_login        TIMESTAMP NULL,
  last_ip           VARCHAR(45),
  reset_token       VARCHAR(255),
  reset_token_expiry TIMESTAMP NULL,
  created_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  updated_at        TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

-- ─────────────────────────────────────────────────────────────
-- TABLE: app_settings
-- Purpose: Global application settings, especially security_mode
-- This is the key table — the toggle switch reads/writes here
-- ─────────────────────────────────────────────────────────────
CREATE TABLE IF NOT EXISTS app_settings (
  id              INT PRIMARY KEY DEFAULT 1,
  security_mode   ENUM('vulnerable','hardened') DEFAULT 'vulnerable',
  app_name        VARCHAR(100) DEFAULT 'VulnLab Pro',
  allow_register  BOOLEAN DEFAULT TRUE,
  maintenance     BOOLEAN DEFAULT FALSE,
  updated_at      TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  updated_by      INT,
  FOREIGN KEY (updated_by) REFERENCES users(id) ON DELETE SET NULL
);

-- ─────────────────────────────────────────────────────────────
-- TABLE: channels
-- Purpose: Slack-like messaging channels
-- ─────────────────────────────────────────────────────────────
CREATE TABLE IF NOT EXISTS channels (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  name        VARCHAR(100) UNIQUE NOT NULL,
  slug        VARCHAR(110) UNIQUE NOT NULL,      -- URL-safe name e.g. "general"
  description TEXT,
  is_private  BOOLEAN DEFAULT FALSE,
  created_by  INT,
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (created_by) REFERENCES users(id) ON DELETE SET NULL
);

-- ─────────────────────────────────────────────────────────────
-- TABLE: channel_members
-- Purpose: Many-to-many: users <-> channels
-- ─────────────────────────────────────────────────────────────
CREATE TABLE IF NOT EXISTS channel_members (
  channel_id  INT NOT NULL,
  user_id     INT NOT NULL,
  role        ENUM('member','moderator','owner') DEFAULT 'member',
  joined_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (channel_id, user_id),
  FOREIGN KEY (channel_id) REFERENCES channels(id) ON DELETE CASCADE,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

-- ─────────────────────────────────────────────────────────────
-- TABLE: messages
-- Purpose: Messages in channels
-- Vuln note: content stored RAW (no escaping) in vuln mode → XSS
-- ─────────────────────────────────────────────────────────────
CREATE TABLE IF NOT EXISTS messages (
  id          INT AUTO_INCREMENT PRIMARY KEY,
  channel_id  INT NOT NULL,
  sender_id   INT NOT NULL,
  content     TEXT NOT NULL,            -- stored raw (XSS) or escaped (hardened)
  file_url    VARCHAR(500),
  file_name   VARCHAR(255),
  file_type   VARCHAR(100),
  reply_to    INT,                       -- thread reply support
  is_deleted  BOOLEAN DEFAULT FALSE,
  is_pinned   BOOLEAN DEFAULT FALSE,
  created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  edited_at   TIMESTAMP NULL,
  FOREIGN KEY (channel_id) REFERENCES channels(id) ON DELETE CASCADE,
  FOREIGN KEY (sender_id) REFERENCES users(id) ON DELETE CASCADE,
  FOREIGN KEY (reply_to) REFERENCES messages(id) ON DELETE SET NULL
);

-- ─────────────────────────────────────────────────────────────

PART 5 — SECURITY MODE TOGGLE: COMPLETE SPECIFICATION
═══════════════════════════════════════════════════════════════
5.1 How the Toggle Works
The security mode is stored in app_settings.security_mode in MySQL.
On every single HTTP request, the securityMode middleware reads this value
from the database (or a 10-second in-memory cache to reduce DB load) and
attaches it to req.securityMode.
Every single route handler, GraphQL resolver, middleware, and service
function that has dual behavior checks req.securityMode or
receives mode as a parameter and branches accordingly.
5.2 The Toggle Endpoint
POST /api/admin/security-mode
Authorization: Bearer <admin_or_sudo_jwt>
Body: { "mode": "vulnerable" | "hardened" }
This updates app_settings, logs the change with who changed it and when,
and emits a Socket.io event security:mode_changed so all connected clients
update their UI in real time.
5.3 Frontend Toggle Component
The toggle is visible in the NAVBAR for all users (read-only for non-admins).
For admins, clicking it shows a confirmation modal:
"You are switching to [VULNERABLE/HARDENED] mode. This affects all users."
Visual design:
  [ ⚠ VULNERABLE MODE ]    [ ✓ HARDENED MODE ]
       (red glow)                (green glow)
It is an animated pill switch, not a checkbox.
When mode changes, a toast notification appears on ALL connected clients
saying "Security mode changed to [mode] by [admin_username]".

HARDENED: Strict allowlist of template names; no user-controlled path components

Clickjacking

VULNERABLE: No X-Frame-Options header → VulnLab Pro can be embedded in iframe
Exploit: Embed login/transfer page in invisible iframe → click-jacking attack
HARDENED: X-Frame-Options: DENY header via Helmet

CORS Misconfiguration (with Credentials)

VULNERABLE: Access-Control-Allow-Origin: * AND Access-Control-Allow-Credentials: true
(This is invalid by spec but some servers allow it improperly)
More practically: CORS set to reflect any Origin header value with credentials
Exploit: Attacker's site at evil.com makes credentialed requests to vulnlab.local; reads responses
HARDENED: CORS origin is a static allowlist from environment variable

XXE (XML External Entity)

Location: POST /api/admin/import accepts XML format
VULNERABLE:

javascript  // [VULN-XXE] External entity processing enabled
  const parser = new xml2js.Parser({ explicitArray: false, allowBeforeBom: true });
  // No XXE protection
Exploit payload:
xml  <?xml version="1.0"?>
  <!DOCTYPE foo [<!ENTITY xxe SYSTEM "file:///etc/passwd">]>
  <users><user>&xxe;</user></users>

HARDENED: Disable external entity processing; use JSON import instead

Race Condition

Location: Balance transfer endpoint
VULNERABLE: Read balance → check sufficient funds → deduct → write (non-atomic)
Two simultaneous requests can both pass the check before either deducts
Exploit: Send 10 concurrent transfer requests; each reads original balance; multiple succeed
HARDENED: Use MySQL transaction with SELECT FOR UPDATE to lock the row during transfer

GraphQL Introspection Abuse

VULNERABLE: Introspection fully enabled → attacker maps entire schema
Exploit: { __schema { types { name fields { name } } } } reveals all types, fields, mutations
HARDENED: Disable introspection in production/hardened mode

GraphQL Batch Attack / Brute Force

VULNERABLE: Multiple operations in single request; no complexity limit
Exploit: Send 1000 login mutations in one request to brute-force passwords
HARDENED: Disable batching; add query complexity limits

// Heuristics for suspicious input detection
function isSuspicious(body, query) {
  const combined = JSON.stringify({ body, query }).toLowerCase();
  const patterns = [
    "' or ", "union select", "1=1", "--", "/*", "*/",
    "<script>", "javascript:", "../", "file://",
    "http://127.", "http://192.168.", "http://10.",
    "whoami", "cat /etc", "/bin/sh", "eval(",
    "base64_decode", "system("
  ];
  return patterns.some(p => combined.includes(p));
}

## Security Mode Explained

### Vulnerable Mode (Default — Red Indicator)

All OWASP Top 10 vulnerabilities are ACTIVE. The application intentionally:
- Uses SQL string concatenation (SQL injection possible)
- Stores passwords as plaintext
- Has no rate limiting on login (brute force possible)
- Returns full user data to any authenticated user (IDOR)
- Renders messages as raw HTML (XSS possible)
- Logs full request bodies including passwords
- Has no CSRF protection
- Accepts any file type for upload

### Hardened Mode (Green Indicator)

All vulnerabilities are MITIGATED:
- Parameterized queries everywhere
- bcrypt password hashing
- Rate limiting (5 attempts per 15 minutes)
- Ownership checks on all data access
- HTML-escaped output (XSS prevented)
- Sensitive fields redacted from logs
- CSRF tokens required
- File type strict allowlisting

### How to Switch Modes

1. Log in as `admin` / `Admin@123`
2. Click the mode toggle in the top navbar (red/green pill)
3. Confirm in the modal
4. Mode changes instantly for ALL users

markdown## A03.1 — SQL Injection: Login Bypass

**Category:** OWASP A03 — Injection
**Severity:** Critical
**Location:** POST /api/auth/login, field: username
**Mode:** Vulnerable only

### Description
The login endpoint directly concatenates user input into a SQL query without
sanitization or parameterization, allowing an attacker to manipulate the query
logic.

### Vulnerable Code
```javascript
// backend/routes/auth.js line ~45
const query = `SELECT * FROM users WHERE username='${username}' AND password='${password}'`;
```

### Exploitation Steps

1. Navigate to http://localhost:3000/login
2. In the username field, enter: `' OR '1'='1' -- `
3. In the password field, enter anything (e.g., `anything`)
4. Click Login
5. The resulting SQL query becomes:
   SELECT * FROM users WHERE username='' OR '1'='1' -- ' AND password='anything'
6. Because '1'='1' is always true and the rest is commented out,
   the query returns the first user in the database (admin)
7. You are now logged in as admin

### Using sqlmap
```bash
sqlmap -u "http://localhost:3000/api/auth/login" \
  --data='{"username":"test","password":"test"}' \
  --method=POST \
  --headers="Content-Type: application/json" \
  --dbs --tables --dump --batch
```

### Expected Result
Full admin access; also extracts all users table data with sqlmap.

### Hardened Fix
```javascript
// Parameterized query — injection impossible
const [rows] = await db.execute(
  `SELECT * FROM users WHERE username = ?`,
  [username]
);
const valid = await bcrypt.compare(password, rows[0]?.password || '');
```

### Prevention Principle
Always use parameterized queries (prepared statements). Never concatenate
user input into SQL strings. Use an allowlist for any dynamic SQL parts
(like column names for ORDER BY).
